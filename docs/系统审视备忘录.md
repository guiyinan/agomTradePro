## 二、P1 级别高优先级问题

### 2.1 模板 URL 硬编码 ⚠️
**影响范围**: 几乎所有模板文件
**问题**: 大量使用硬编码 URL 而非 `{% url %}` 标签

**示例**:
| 文件 | 行号 | 问题 |
|------|------|------|
| `base.html` | 19, 21, 28-34, 43-45 | 导航链接硬编码 |
| `base.html` | 54-57, 64-65, 77-80 | 系统菜单硬编码 |
| `base.html` | 84-85 | 登录/注册链接硬编码 |
| `regime/dashboard.html` | 65 | API 路径硬编码 |
| `signal/manage.html` | 405, 431, 445, 454, 469, 478 | 多处硬编码 |

**风险**: URL 配置变更时产生 404 错误

**修复方案**: 全部改为 `{% url 'app_name:view_name' %}` 格式

---

### 2.2 JavaScript API 路径硬编码 ⚠️
**影响**: AJAX 请求在路由变更时失效

**示例**:
| 文件 | 位置 | 硬编码路径 |
|------|------|-----------|
| `regime/dashboard.html` | 第 65 行 | `fetch('/macro/api/quick-sync/')` |
| `signal/manage.html` | 第 269 行 | `fetch('/signal/ai/parse-logic/')` |
| `filter/dashboard.html` | 第 41 行 | `fetch('/filter/api/')` |

**修复方案**: 使用 `data-url` 属性或 Django 的 `{% url %}` 配合 JavaScript

---

## 四、URL 路由规范问题

### 4.1 API 路径前缀不统一
**示例**:
- Account: `/account/api/portfolios/` ✓ 标准
- Backtest: `/backtest/api/backtests/` ✗ 嵌套重复
- Macro: `/macro/api/fetch/` ✗ 直接暴露
- Signal: `/signal/api/` ✓ 标准

**建议规范**:
```
页面路由: /app_name/page_name/
API 路由: /app_name/api/resource_name/
DRF 路由: /app_name/api/resource_name/{id}/action_name/
```

---

### 4.2 DRF Router 和普通 API 混用
**问题**: Backtest 模块同时使用 DRF ViewSet 和普通视图函数

**示例**:
- `/backtest/api/run/` (普通视图)
- `/backtest/api/backtests/` (DRF ViewSet)

**建议**: 统一使用 DRF ViewSet

---

### 4.3 Regime 模块空路由前缀
**位置**: `apps/regime/interface/urls.py`
**问题**: `router.register(r'', RegimeViewSet, basename='regime')`

**风险**: 在 `/regime/api/` 下注册空字符串前缀，可能导致根路由冲突

## 九、总结

### 系统整体评分
- **核心架构**: A+ (严格的四层架构，依赖注入完善)
- **数据完整性**: B (Macro 模块断点拉低分数)
- **用户体验**: B- (硬编码 URL，错误处理不统一)
- **功能实现度**: A- (大部分功能完整，少数非关键功能待实现)

### 最关键的修复项
1. **Macro 模块 views.py** - 影响整个宏观数据采集功能
2. **ECharts 库** - 影响 Regime 判定页面展示
3. **URL 标签化** - 提高系统可维护性

### 系统可用性结论
🟡 **部分可用** - 核心功能（Regime、Policy、Signal、Backtest）完整，但 Macro 模块缺失导致数据采集功能完全不可用，这是整个系统的上游断点，必须优先修复。
