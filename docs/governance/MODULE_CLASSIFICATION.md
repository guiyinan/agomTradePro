## 1. 分级原则

模块分级基于以下维度：

1. **被依赖次数**: 越多依赖，越核心
2. **稳定性要求**: 接口变更影响面
3. **测试覆盖**: 测试完整度
4. **功能成熟度**: 生产使用情况

---

## 2. 三级分类

### 🔴 核心内核（Tier 1）- 不能乱动

**定义**: 被大量模块依赖，接口变更影响面极大

**数量**: 4 个

**变更规则**:
- ❌ 禁止破坏性接口变更
- ✅ 只允许向后兼容的扩展
- ✅ 变更必须经过架构评审
- ✅ 变更前必须通知所有依赖方

| 模块 | 职责 | 被依赖次数 | 关键性 |

| **regime** | 宏观象限判定引擎 | 10 | 🔴 极高 |
| **ai_provider** | AI 服务商管理 | 5 | 🔴 极高 |
| **events** | 事件发布与订阅 | 4 | 🔴 高 |
| **macro** | 宏观数据采集 | 4 | 🔴 高 |


## 5. 变更影响评估

### 5.1 变更影响矩阵

| 变更模块 | 影响范围 | 必须通知 | 评审级别 |
|----------|----------|----------|----------|
| **Tier 1** | 全局 | 所有依赖方 | 架构评审 |
| **Tier 2** | 局部 | 直接依赖方 | 技术评审 |
| **Tier 3** | 自身 | 无 | 代码评审 |

### 5.2 破坏性变更定义

以下变更视为**破坏性变更**：

- ❌ 删除公开 API 端点
- ❌ 修改 API 参数签名（删除/重命名必填参数）
- ❌ 修改返回值结构（删除字段）
- ❌ 修改 Domain 层实体定义
- ❌ 修改数据库 Schema（非加列）

## 6. 升级/降级规则

### 6.1 升级到 Tier 2

**条件**（需同时满足）：

1. ✅ 被依赖次数 ≥ 2
2. ✅ 测试覆盖率 ≥ 85%
3. ✅ 生产环境使用 ≥ 3 个月
4. ✅ 接口稳定 ≥ 2 个月无破坏性变更

### 6.2 升级到 Tier 1

**条件**（需同时满足）：

1. ✅ 被依赖次数 ≥ 5
2. ✅ 测试覆盖率 ≥ 90%
3. ✅ 生产环境使用 ≥ 6 个月
4. ✅ 接口稳定 ≥ 6 个月无破坏性变更
5. ✅ 架构评审通过

### 6.3 降级

**触发条件**：

- ⚠️ 连续 3 次破坏性变更
- ⚠️ 测试覆盖率 < 80%
- ⚠️ 生产环境重大故障

## 7. 监控与告警

### 7.1 依赖监控

- **监控指标**: 模块依赖数、被依赖数
- **告警阈值**: 依赖数 > 10 或被依赖数 > 5 时告警
- **检查频率**: 每次发布前

### 7.2 测试监控

- **监控指标**: 测试覆盖率、测试通过率
- **告警阈值**: Domain 层覆盖率 < 90%
- **检查频率**: 每次 PR
