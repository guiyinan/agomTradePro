## 三、问题清单（按优先级）

### P1: 登录态访问文档管理/详情页 500

- 现象:
  - `GET /admin/docs/manage/` -> 500
  - `GET /docs/uat-doc-json/`（登录态）-> 500
- 错误:
  - `NoReverseMatch: Reverse for 'token_management' not found`
- 初步定位:
  - `core/templates/base.html` 使用了 `account:token_management` / `account:user_management`
  - 但 `core/urls.py` 当前对 account 的 include 未声明 namespace（`path('account/', include('apps.account.interface.urls'))`）
- 影响:
  - 管理员登录后进入文档相关页面失败，阻断后台文档运营流程。

### P1: SDK `signal.check_eligibility` 稳定失败

- 现象:
  - SDK 调用 `signal/api/check_eligibility/` 连续返回 500，触发重试后失败。
- 影响:
  - 基于 SDK 的资格校验流程不可用。

### P2: SDK `regime.history` 返回结构兼容问题

- 现象:
  - `regime.history()` 解析时假设响应对象支持 `.get()`，实际返回 list，导致异常。
- 影响:
  - SDK 在部分环境下无法读取历史 Regime。

### P3: 宏观数据创建接口对非法参数返回 500（可优化为 400）

- 现象:
  - `/macro/api/record/create/` 缺少 `code` 时由 DB 约束抛错，返回 500。
- 影响:
  - 用户输入错误时提示不友好；不利于前端做精确提示。

## 四、导入能力验收结论

- 宏观数据导入: 通过（手动新增 + 快速同步均可）
- 文档导入（JSON/CSV）: 导入能力本身通过（创建/更新成功，数据已落库）
- 文档导入后可视化验证: 受 P1 路由问题影响，登录态页面验证失败

## 五、建议的修复顺序

1. 修复 account URL namespace 与模板反向解析不一致问题（先恢复文档管理可用性）。
2. 修复 `signal.check_eligibility` 后端 500（查看接口日志和输入校验链路）。
3. 修复 SDK `regime.history` 对 list/dict 的兼容解析。
4. 将宏观创建接口参数错误改为 400 级业务错误返回。

## 十、系统 RBAC 与 MCP 对齐（2026-02-07）

目标：让“系统用户角色”与 “MCP RBAC”使用同一套角色语义与权限矩阵，避免双轨配置漂移。

实施内容：

1. 后端统一 RBAC 模型与矩阵
- 新增统一 RBAC 逻辑：`apps/account/application/rbac.py`
- 角色：`admin/owner/analyst/investment_manager/trader/risk/read_only`
- 新增账户角色字段：`AccountProfileModel.rbac_role`
  - 模型：`apps/account/infrastructure/models.py`
  - 迁移：`apps/account/migrations/0013_accountprofilemodel_rbac_role.py`
  - 数据迁移策略：`superuser -> admin`，其余用户 -> `owner`

2. 系统侧权限接入统一角色
- 管理员页面权限由“仅 superuser”升级为“系统 RBAC admin”判定：
  - `apps/account/interface/views.py` (`is_admin_user -> is_system_admin`)
- 用户注册时同步初始化 `rbac_role`：
  - 首个自动管理员：`admin`
  - 其余用户：`owner`
- 新增管理员设置用户角色接口：
  - `POST /account/admin/users/<user_id>/role/`
  - 路由：`apps/account/interface/urls.py`

3. Account API 按 RBAC 矩阵限权
- 新增 DRF 权限类：`apps/account/interface/permissions.py`
- 交易域接口（组合/持仓/交易/资金流水）接入 `TradingPermission`
- 账户配置与健康检查接入 `GeneralPermission`
- 结果：系统 API 写操作将按角色矩阵限制，不再只依赖“是否已登录”

4. MCP 与系统角色自动对齐
- MCP RBAC 默认从后端用户资料读取 `rbac_role`（`account/api/profile/`）
- `AGOMSAAF_MCP_ROLE` 仍可作为强制覆盖
- 文档更新：`sdk/docs/mcp_guide.md`

5. 验证结果
- `pytest -q sdk/tests/test_mcp/test_rbac.py` -> `22 passed`
- `pytest -q tests/acceptance/test_sdk_connection.py` -> `8 passed, 2 skipped`
- `pytest -q tests/acceptance/test_mcp_server.py` -> `9 passed`
- 迁移执行：`python manage.py migrate` -> `account.0013 ... OK`

结论：
- 系统用户角色与 MCP RBAC 已对齐到同一角色体系。
- 后端角色变更可直接驱动 MCP 权限，无需重复维护两套角色配置。
